## E2E findings

### H-E1 (HIGH) — Library import fails on plain Node 22+ from `node_modules`
**Root cause:** Published `package.json` has `"main": "src/index.ts"`. Node 22+'s built-in type-stripping (`--experimental-strip-types`) refuses to strip `.ts` files under `node_modules`:
```
Error [ERR_UNSUPPORTED_NODE_MODULES_TYPE_STRIPPING]: Stripping types is currently
unsupported for files under node_modules, for "file:///.../node_modules/@shipispec/tsfix/src/index.ts"
```
Consumer must use a loader (`tsx`, `ts-node`, `jiti`) or a TS-aware bundler. Plain `node` doesn't work.

**Repro:** `mkdir x && cd x && npm init -y && npm install @shipispec/tsfix typescript && echo 'import {runValidationLoop} from "@shipispec/tsfix"; console.log(typeof runValidationLoop);' > t.mjs && node t.mjs`

**Severity rationale:** Most JS-only consumers will hit this immediately. tsx-equipped consumers are unaffected.

**Fix:** Phase 1a (esbuild bundle producing `dist/index.js` + `dist/index.d.ts`; repoint `main`/`types`/`exports`).

**Target:** `0.2.0` (Phase 1a).

### M-E1 (MEDIUM) — Bin errors with stale name string
**Root cause:** `bin/tsfix.mjs` in the published 0.1.0 tarball still says `"error: tsc-defense requires \`tsx\`..."` — pre-rename text. Working tree fixed in commit `b502f00`+, will land in `0.1.1`.

**Repro:** `npx @shipispec/tsfix --help` (cold) → outputs the wrong product name.

**Fix:** Already committed. Just publish `0.1.1`.

**Target:** `0.1.1`.

### M-E2 (MEDIUM) — `npx @shipispec/tsfix` cold-start fails
**Root cause:** `bin/tsfix.mjs` requires `tsx` in the package's own `node_modules`. `npx` against a published tarball doesn't install our devDeps. Same root as H-E1.

**Repro:** `npx @shipispec/tsfix --help` from any directory.

**Fix:** Phase 1a.

**Target:** `0.2.0`.

### M-E3 (MEDIUM) — `npm scripts` ship to consumers but reference missing files/deps
The published `package.json` includes `scripts.benchmark` (`tsx benchmark/run-benchmark.ts`), `scripts.test` (`vitest run`), `scripts.setup-fixtures` (references `fixtures/_shared/`). None of `tsx`, `vitest`, or `fixtures/` ships in the tarball. A consumer running `npm run benchmark` from `node_modules/@shipispec/tsfix/` gets a confusing "command not found" error.

**Fix options:**
- (a) Strip dev-only scripts from the published `package.json` (use a `prepare` step + `clean-publish` package, or maintain two package.json files)
- (b) Document that scripts are dev-only, intended for contributors not consumers
- (c) Leave as-is — npm packages routinely ship dev scripts with no consumer harm

Lean (b): one-line README note.

**Target:** `0.1.1` (docs only) or `0.2.0` (if (a) chosen).

### M-E4 (MEDIUM) — Footgun: bin/CLI mutates files in place with no `--dry-run`
Already documented in `internal/STATUS.md` and `../CHANGELOG.md`. The CLI persists LSP edits to disk. Pointing it at a fixture directory irreversibly fixes the broken code (we hit this during Phase 0c verification).

**Fix:** Add `--dry-run` flag to CLI; print would-be edits without writing.

**Target:** `0.1.1` (small change, high user value).

### M-S1 (MEDIUM) — README lacks workspace-trust disclosure
**Issue:** The package loads `typescript` from the consumer's `node_modules/typescript` (per the lib-path override that makes esbuild bundling work). If the workspace has a malicious `typescript` install, that's arbitrary code execution at validate time. This is normal for any tool that loads the workspace's TypeScript (eslint, prettier, etc. all do this), but the OSS audience deserves an explicit statement.

**Fix:** Add to the published README:
> **Trust model:** This tool loads `typescript` from the workspace's `node_modules`. Only run on workspaces you trust.

Roadmap M2 already lists this; just hadn't made it into the README yet.

**Target:** `0.1.1`.

### L-S1 (LOW) — Missing npm provenance
**Issue:** v0.1.0 was published manually, not from CI. npm 9.5+ supports `--provenance` to sign publishes via GitHub Actions OIDC, providing supply-chain attestation ("this tarball really came from this commit on main via this workflow"). Without it, no cryptographic proof of origin.

**Fix:** Phase 1b CI workflow uses `permissions: id-token: write` and `npm publish --provenance --access public`.

**Target:** `0.2.0` (Phase 1b — first CI-published release).

### L-S2 (LOW) — `.ts` files in tarball have exec bits
The published `cli/run-stack.ts` is `-rwxr-xr-x`. It's loaded by `tsx`, never run directly. The exec bit is harmless but slightly confusing — tarball auditors may wonder why a `.ts` file is executable.

**Fix:** `chmod 644 cli/run-stack.ts` before the next pack. Or set the file-mode in source control with `git update-index --chmod=-x`.

**Target:** `0.1.1`.

### L-P1 (LOW polish) — `engines.npm` not declared
We pin `engines.node: ">=20.9.0"` but not npm. Some package managers (yarn, pnpm) honor `engines.node` differently than npm. Adding `engines.npm: ">=10.0.0"` makes the requirement explicit. Low priority.

**Target:** `0.1.1` or later.
